Tutor Support System — Class Diagram Description (UML-Ready Version)
1. Overview
The class diagram models the structural design of the Tutor Support System (TSS). It identifies the main classes, their attributes, operations, package groupings, and inter-class relationships. The model supports authentication, academic data management, scheduling and matching, session handling, progress tracking, reporting, and external data integration.
2. Package Structure
Identity & Access
Users — SSO identity; minimal authentication profile.
Role — role catalog.
RoleAssignment — association class linking Users ↔ Roles.
AuditLog — append-only audit records.
Academic Master Data
Department — department list.
Course — course offerings linked to Term and Department.
Term — academic periods.
Enrollment — student–course registration.
Scheduling & Sessions
AvailabilitySlot — tutor availability windows.
Booking — student reservation of a slot.
Session — confirmed meeting derived from a Booking.
MatchingRule — criteria for recommending tutors.
MatchingService — ranking and match selection engine.
NotificationService — booking/session communication.
Progress, Feedback & Reporting
ProgressLog — tutor notes per session.
Attachment — file metadata linked to logs/sessions.
Feedback — student evaluations.
ReportDefinition — report templates and parameters.
ReportRun — report execution results.
Integration & Data
DataSyncJob — scheduled ETL processes.
ExternalClients — interfaces to SSO, DataCore, Library.
Repository — aggregation of persistence interfaces.

Users
Attributes: userId, email, displayName, status.

Department
Attributes: deptId, name.

Attributes: termId, name, startDate, endDate.

Enrollment

4. Relationships (UML-Oriented)
Users 1..* ↔ *..1 Role via RoleAssignment.
Users (Student) 1..* ↔ *..1 Enrollment ↔ 1..1 Course.
Users (Tutor) 1..1 ↔ 0..* AvailabilitySlot.
AvailabilitySlot 1..1 ↔ 0..1 Booking.
Booking 1..1 → 1..1 Session.
Session 1..1 → 0..* ProgressLog; 1..1 → 0..* Feedback.
ProgressLog 1..1 → 0..* Attachment.
ReportDefinition 1..1 → 0..* ReportRun.
DataSyncJob → ExternalClients, updates Repository.
AuditLog receives actions from all packages.
5. Constraints and Invariants
Availability slots for a tutor must not overlap.
Booking confirmation only when slot is in “held” state.
Session rescheduling and cancellation follow configured time windows.
Feedback editable only within a defined post-session window.
ReportRun objects immutable after completion.
All privileged actions must generate an AuditLog entry.
6. Use Case Traceability
Login
Users authenticate via SSO → Roles loaded → AuditLog updated.
Booking Workflow
Student views AvailabilitySlot → creates Booking → confirms → Session created → notifications sent.
Session Workflow
Tutor records attendance → adds ProgressLog → uploads Attachments → Student submits Feedback.
Reporting
Reports generated from ReportDefinition → output recorded in ReportRun.
Synchronization
DataSyncJob pulls master data → updates repositories → logs reconciliation actions.
